# docs/adr

## 役割

このディレクトリには、重要な技術判断を Architecture Decision Record (ADR) として記録します。

## ADR とは

Architecture Decision Record は、システム設計における重要な決定とその理由を記録するドキュメントです。

### ADR の対象

以下のような重要な技術判断が対象となります：

- 使用する言語やフレームワーク
- データベース選定（RDBMS, NoSQL など）
- 認証・認可方式
- API 方式（REST, GraphQL など）
- リポジトリ構成
- 外部サービスの採用
- キャッシング戦略
- ログ戦略
- 後から変更しにくい設計判断

### ADR の価値

- **意思決定の記録**: なぜその判断をしたのかを将来のチームメンバーが理解できる
- **代替案の検討**: なぜ他の選択肢を選ばなかったのかが明確
- **変更時の判断**: 後から判断を見直す際、前提条件の変化を認識できる

## ファイル構成

```
docs/adr/
├── README.md (このファイル)
├── 0001-use-python.md
├── 0002-use-postgresql.md
├── 0003-rest-api.md
└── ...
```

各ファイルは、1 つの重要な技術判断を記録します。

## 命名規則

- ファイル名は `NNNN-[タイトル].md` の形式
- `NNNN` は 4 桁のゼロパディングされた連番
- 例：`0001-use-python.md`, `0002-use-postgresql.md`

## テンプレート

新しい ADR を作成する際は、以下のテンプレートを使用してください：

```markdown
# ADR-NNNN: [タイトル]

**ステータス**: [Proposed | Accepted | Deprecated | Superseded]

**日付**: YYYY-MM-DD

## 背景

[判断が必要になった背景とコンテキストを記載]

## 問題

[解決すべき問題を記載]

## 決定

[下した決定を明確に記載]

## 理由

[その判断に至った理由を詳細に記載]

## 代替案

| 代替案 | 利点 | 欠点 |
|------|------|------|
| [代替案 1] | [利点] | [欠点] |
| [代替案 2] | [利点] | [欠点] |

## 結果

### ポジティブな影響

- [ポジティブな影響]

### ネガティブな影響

- [ネガティブな影響]

### リスク

- [リスク]

## 関連資料

- [参考資料 1]
- [参考資料 2]

## 追記

[後の段階で追記があれば記載]
```

## ステータスの定義

| ステータス | 説明 |
|----------|------|
| Proposed | 提案段階 |
| Accepted | 承認済み |
| Deprecated | 非推奨（新規では使用しない） |
| Superseded | 置き換わった（新しい ADR により置き換わった） |

## ADR 作成時の注意点

- **簡潔さ**: 重要な内容に絞る
- **客観性**: 恣意的でなく、事実とロジックに基づく
- **未来志向**: 将来のメンバーが理解できるように
- **トレーサビリティ**: 関連する設計書や実装コードへの参照

## 例

### 例 1: フレームワーク選定

```markdown
# ADR-0001: Use Django for Web Framework

**ステータス**: Accepted

**日付**: 2026-01-15

## 背景

新しい Web アプリケーションを開発する必要があり、Python ベースの Web フレームワークを検討している。

## 問題

複数の Web フレームワーク（Django, FastAPI, Flask）から 1 つを選定する必要がある。

## 決定

Django を採用する。

## 理由

- 豊富なドキュメントと大規模なコミュニティ
- ORM が充実しており、開発効率が高い
- 認証・認可機能が組み込まれている
- 管理画面が自動生成される

## 代替案

| 代替案 | 利点 | 欠点 |
|------|------|------|
| FastAPI | モダンな設計、高速 | ORM 機能が限定的 |
| Flask | 軽量 | 基礎機能を自分で実装する必要がある |

## 結果

### ポジティブな影響

- 開発速度が向上
- チームの学習コストが低い

### ネガティブな影響

- フレームワーク的な制約がある

## 関連資料

- Django 公式ドキュメント: https://docs.djangoproject.com/
```
