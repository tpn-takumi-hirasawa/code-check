# Development Workflow

Claude Code を使用した開発フローの詳細手順を定義します。

## 開発フロー全体

```
要求整理
  ↓
仕様書作成
  ↓
仕様レビュー
  ↓
設計書作成
  ↓
設計レビュー
  ↓
実装計画
  ↓
実装
  ↓
コードレビュー
  ↓
テスト
  ↓
ドキュメント更新
  ↓
Pull Request 作成
```

## 各工程の詳細

### 1. 要求整理

**担当**: `requirements-analyst` エージェント

**入力**:
- 要求や仕様に関する説明
- ステークホルダーからのフィードバック

**出力**:
- 要求の整理結果
- 目的、利用者、機能要件の明確化
- 非機能要件、制約、前提の記録
- 対象外と未決事項の列挙

**検証**:
- 利害関係者による確認
- 不明な点は TBD として記録

**チェックリスト**:
- [ ] 目的が明確
- [ ] 利用者が定義されている
- [ ] 機能要件が列挙されている
- [ ] 非機能要件が列挙されている
- [ ] 制約が記録されている
- [ ] 前提が記録されている
- [ ] 対象外が明確
- [ ] 未決事項が列挙されている

### 2. 仕様書作成

**担当**: `spec-writer` エージェント

**入力**:
- 整理済みの要求
- プロジェクト概要

**出力**:
- 機能仕様書（`docs/specs/` 配下）

**仕様書に含める内容**:
- 目的
- 対象範囲
- 利用者・アクター
- 機能要件（FR-XXX ID付き）
- 業務ルール（BR-XXX ID付き）
- 正常フロー
- 例外フロー
- 入力検証
- 権限
- エラー処理
- 受入条件（AC-XXX ID付き）
- 対象外
- 未決事項

**検証**:
- 実装者による可読性確認
- テスト計画者による完全性確認
- 利害関係者による正確性確認

**チェックリスト**:
- [ ] すべてのファイルに ID（FR, BR, AC）が付与されている
- [ ] 受入条件が SMART である
- [ ] 未決事項が明確
- [ ] テスト計画者が確認済み

### 3. 仕様レビュー

**担当**: 利害関係者

**活動**:
- 仕様書の正確性確認
- 要件の完全性確認
- 用語の一致確認

**出力**:
- 承認またはコメント

**チェックリスト**:
- [ ] すべての要件が仕様書に記載されている
- [ ] 要件が正確に表現されている
- [ ] テスト可能な受入条件が定義されている
- [ ] 利害関係者が承認している

### 4. 設計書作成

**担当**: `design-writer` エージェント

**入力**:
- 承認済みの仕様書
- 既存システムアーキテクチャ

**出力**:
- 技術設計書（`docs/design/` 配下）

**設計書に含める内容**:
- 設計概要
- アーキテクチャ
- コンポーネント構成
- データモデル
- API
- 処理フロー
- エラー処理
- 認証・認可
- セキュリティ
- 性能
- ログ・監視
- テスト方針
- 移行方針
- リスク

**制約**:
- 仕様に存在しない機能を追加しない
- 設計判断の理由を記録する

**検証**:
- 実装者による実現可能性確認
- アーキテクト による整合性確認

**チェックリスト**:
- [ ] 仕様のすべての要件が設計で対応されている
- [ ] アーキテクチャが明確
- [ ] コンポーネント間の依存関係が明確
- [ ] データモデルが定義されている
- [ ] API 仕様が明確
- [ ] テスト計画が定義されている

### 5. 設計レビュー

**担当**: アーキテクト

**活動**:
- 設計の整合性確認
- 既存システムとの互換性確認
- 性能と スケーラビリティ確認

**出力**:
- 承認またはコメント

**チェックリスト**:
- [ ] 設計が仕様に適合している
- [ ] アーキテクチャが適切
- [ ] リスクが軽減されている
- [ ] アーキテクトが承認している

### 6. 実装計画

**担当**: `implementation-planner` エージェント

**入力**:
- 承認済みの仕様書
- 承認済みの設計書
- 既存コード

**出力**:
- 実装計画

**計画に含める内容**:
- 作成するファイル
- 変更するファイル
- 実装順序
- 完了条件
- 必要なテスト
- 検証コマンド
- リスク

**チェックリスト**:
- [ ] ファイル構成が明確
- [ ] 実装順序が依存関係を考慮している
- [ ] 完了条件が明確
- [ ] テスト方針が定義されている

### 7. 実装

**担当**: 実装者

**入力**:
- 実装計画

**出力**:
- コード（GitHub ブランチ）

**ガイドライン**:
- 実装計画に従う
- テスト可能な設計を心がける
- コメントは必要最小限

**チェックリスト**:
- [ ] コードが実装計画に従っている
- [ ] ビルドが成功
- [ ] テストが実装されている
- [ ] ドキュメント が更新されている

### 8. コードレビュー

**担当**: `code-reviewer` エージェント

**入力**:
- 現在のブランチとメインブランチの差分

**出力**:
- レビュー結果（BLOCKER, HIGH, MEDIUM, LOW, SUGGESTION）

**確認項目**:
- 仕様適合性
- 設計適合性
- 正しさ
- 可読性
- 保守性
- エラー処理
- セキュリティ
- 性能
- 後方互換性
- テスト不足

**チェックリスト**:
- [ ] BLOCKER 問題がない
- [ ] HIGH 以上の問題が解決されている
- [ ] テストが実装されている
- [ ] ドキュメントが更新されている

### 9. テスト

**担当**: `test-writer` エージェント

**入力**:
- 仕様書の受入条件
- 実装コード

**出力**:
- テストコード

**確認項目**:
- 正常系テスト
- 異常系テスト
- 境界値テスト
- 入力検証テスト
- 権限テスト
- エラー処理テスト

**チェックリスト**:
- [ ] 受入条件がすべてテストされている
- [ ] テストが実行でき、成功している
- [ ] カバレッジが目標値を満たしている
- [ ] テストが自動化されている

### 10. ドキュメント更新

**担当**: `docs-maintainer` エージェント

**入力**:
- 実装コード
- 仕様書
- 設計書

**出力**:
- 更新されたドキュメント

**確認項目**:
- 仕様とコードの一致
- 設計とコードの一致
- API ドキュメント
- 使用方法ドキュメント

**チェックリスト**:
- [ ] ドキュメントとコードが同期している
- [ ] API ドキュメント が正確
- [ ] 使用例が含まれている

### 11. Pull Request 作成

**担当**: `pr-writer` エージェント

**入力**:
- コミット履歴
- レビュー結果
- テスト結果

**出力**:
- Pull Request

**PR に含める内容**:
- 変更の概要
- 関連する仕様・設計への参照
- テスト結果
- 性能への影響
- 既知の制限事項

**チェックリスト**:
- [ ] PR タイトルが明確
- [ ] PR 説明が明確
- [ ] テスト結果がすべて成功
- [ ] コードレビュー が完了
- [ ] リスクが軽減されている

## Pull Request の推奨単位

```
PR 1: リポジトリ初期構成
PR 2: CLAUDE.md と開発ルール
PR 3: カスタムエージェント
PR 4: プロジェクト概要
PR 5: 機能仕様書
PR 6: 技術設計書・ADR
PR 7: 実装
PR 8: テスト・ドキュメント更新
```

変更規模が小さい場合は統合しても構いませんが、仕様、設計、実装を一つの巨大な Pull Request にまとめないことを推奨します。

## ロール

| ロール | 責務 |
|------|------|
| 要求アナリスト | 要求の整理と明確化 |
| 仕様ライター | 機能仕様書の作成 |
| 設計ライター | 技術設計書の作成 |
| 実装者 | コードの実装 |
| コードレビュアー | コード品質のレビュー |
| テストライター | テストコードの作成 |
| セキュリティレビュアー | セキュリティ面でのレビュー |
| ドキュメンテーション担当者 | ドキュメント品質の確認 |
