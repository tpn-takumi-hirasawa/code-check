# docs/specs

## 役割

このディレクトリには、システムや機能が「何をするか」を定義する仕様書を保存します。

## 仕様書の構成

各仕様書は、以下の内容を含むべきです：

- **目的**: 機能の目的
- **対象範囲**: 仕様が適用される範囲
- **利用者・アクター**: 機能を利用する者
- **機能要件（FR-XXX）**: システムが満たすべき機能（ID付き）
- **業務ルール（BR-XXX）**: 業務上のルール（ID付き）
- **正常フロー**: 通常の処理フロー
- **例外フロー**: エラー時の処理フロー
- **入力検証**: 入力値の検証ルール
- **権限**: アクセス制御に関する要件
- **エラー処理**: エラー発生時の対応
- **受入条件（AC-XXX）**: テスト可能な完成条件（ID付き）
- **対象外**: 仕様の対象外範囲
- **未決事項**: 確定していない項目（TBD）

## ファイル構成

```
docs/specs/
├── README.md (このファイル)
├── feature-1.md
├── feature-2.md
└── ...
```

各ファイルは、1 つの機能または 1 つの機能グループを定義します。

## 命名規則

- ファイル名は英数字とハイフン（`-`）のみを使用
- 例：`user-authentication.md`, `payment-processing.md`

## テンプレート

新しい仕様書を作成する際は、以下のテンプレートを使用してください：

```markdown
# [機能名] - Specification

## 目的

[機能の目的と背景を記載]

## 対象範囲

[仕様が適用される範囲を記載]

## 利用者・アクター

| アクター | 説明 |
|---------|------|
| [アクター名] | [説明] |

## 機能要件

| ID | 要件 | 説明 |
|----|------|------|
| FR-001 | [要件] | [説明] |

## 業務ルール

| ID | ルール | 説明 |
|----|--------|------|
| BR-001 | [ルール] | [説明] |

## 正常フロー

[正常時の処理フローを記載]

## 例外フロー

[エラー時の処理フローを記載]

## 入力検証

[入力値の検証ルールを記載]

## 権限

[アクセス制御に関する要件を記載]

## エラー処理

[エラー発生時の対応を記載]

## 受入条件

| ID | 条件 | 説明 |
|----|------|------|
| AC-001 | [条件] | [説明] |

## 対象外

[仕様の対象外範囲を記載]

## 未決事項

- TBD: [未決定事項]
```
