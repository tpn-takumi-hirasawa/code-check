# docs/design

## 役割

このディレクトリには、仕様を「どのように実現するか」を定義する技術設計書を保存します。

## 設計書の構成

各設計書は、以下の内容を含むべきです：

- **設計概要**: 設計の全体像
- **アーキテクチャ**: システム構成とコンポーネント間の関係
- **コンポーネント構成**: 各コンポーネントの責務と依存関係
- **データモデル**: エンティティ、属性、関係
- **API**: エンドポイント、リクエスト、レスポンス
- **処理フロー**: シーケンス図、状態遷移図
- **エラー処理**: エラーハンドリング戦略
- **認証・認可**: セキュリティメカニズム
- **セキュリティ**: 脅威モデル、対策
- **性能**: スケーラビリティ、キャッシング戦略
- **ログ・監視**: ログレベル、メトリクス
- **テスト方針**: テスト戦略、テストケース
- **移行方針**: 既存システムからの移行方法
- **リスク**: 技術的なリスクと対策

## ファイル構成

```
docs/design/
├── README.md (このファイル)
├── feature-1-design.md
├── feature-2-design.md
└── ...
```

各ファイルは、対応する仕様書の技術設計を定義します。

## 命名規則

- ファイル名は英数字とハイフン（`-`）のみを使用
- 対応する仕様書ファイル名に `-design` を追加
- 例：仕様書 `user-authentication.md` → 設計書 `user-authentication-design.md`

## 設計書作成時の注意点

- **仕様との整合性**: 仕様書に存在しない機能を追加しない
- **実装可能性**: 実装チームが現実的に実現できる設計
- **トレーサビリティ**: 仕様書の各要件に対応する設計要素を明確にする
- **拡張性**: 将来の拡張を念頭に置いた設計
- **ドキュメント性**: 図表や例を活用して理解しやすく
