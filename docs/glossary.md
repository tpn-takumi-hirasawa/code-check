# Glossary

このプロジェクトで使用される重要な用語とドメイン概念を定義します。

## プロジェクト関連用語

### Claude Code

Anthropic の AI アシスタント Claude をベースにした、ソフトウェア開発支援ツール。ブラウザ版、デスクトップ版、IDE 拡張版がある。

### カスタムエージェント

Claude Code 内で定義された、特定の役割を担当する専門的な AI エージェント。

例：
- `requirements-analyst`: 要求を整理するエージェント
- `spec-writer`: 仕様書を作成するエージェント
- `code-reviewer`: コードをレビューするエージェント

### スキル

複数の手順をまとめた、再利用可能な作業フロー。

例：
- `create-feature`: 新機能を作成するスキル
- `fix-bug`: バグを修正するスキル

### ルール

特定の領域における開発ガイドラインやベストプラクティス。

例：
- `frontend.md`: フロントエンド開発ルール
- `backend.md`: バックエンド開発ルール

## 開発フロー関連用語

### 要求（Requirement）

ステークホルダーが求める機能、特性、制約条件。

### 仕様（Specification）

要求を詳細に記述したドキュメント。「何をするか」を定義。

### 設計（Design）

仕様を実現するための技術的な方法。「どのように実現するか」を定義。

### 実装計画（Implementation Plan）

実装の対象、順序、方法、テスト方針などを含む計画。

### コードレビュー（Code Review）

実装されたコードの品質、正確性、保守性などを複数の観点から確認すること。

### 受入条件（Acceptance Criteria, AC）

機能が完成したと判定するための条件。テスト可能である必要がある。

### TBD (To Be Determined)

未決定の項目や、確定していない情報に付与するマーク。推測ではなく、意思決定後に確定させるべき項目を示す。

## ドキュメント関連用語

### Architecture Decision Record (ADR)

重要な技術判断とその理由を記録するドキュメント。

### PR（Pull Request）

GitHub における変更提案のこと。通常、レビューと承認を経て統合される。

### CLAUDE.md

Claude Code がプロジェクト内で従う共通ルール、コーディング規約、開発手順などを記載するドキュメント。

### Project Brief

プロジェクトの目的、スコープ、成功基準などを記載するドキュメント。

### Development Workflow

開発の各工程の手順、チェックリスト、成功基準を記載するドキュメント。

## コード品質関連用語

### 仕様適合性

実装が仕様書の要件を満たしているか。

### 設計適合性

実装が設計書の方針に従っているか。

### 正しさ（Correctness）

コードが意図したとおりに動作するか。

### 可読性（Readability）

コードが理解しやすいか。

### 保守性（Maintainability）

コードが変更・拡張しやすいか。

### セキュリティ

コードが認証、認可、入力検証などのセキュリティ要件を満たしているか。

### 性能（Performance）

コードがレスポンス時間やスループットの非機能要件を満たしているか。

### テストカバレッジ

実装されたコードのうち、テストで検証されている割合。

## レビュー結果分類

### BLOCKER

修正が必須であり、これなしにマージできない問題。

### HIGH

修正が強く推奨される重大な問題。

### MEDIUM

修正が推奨される中程度の問題。

### LOW

軽微な問題で、必須ではないが改善が望ましい。

### SUGGESTION

改善提案や参考情報。必須ではない。

## テスト関連用語

### 正常系テスト

想定された入力に対して、期待される出力が得られるかを確認するテスト。

### 異常系テスト

想定外の入力やエラー状態に対して、適切に処理するかを確認するテスト。

### 境界値テスト

入力値の最小値、最大値、その前後における動作を確認するテスト。

### 単体テスト

個別の関数やメソッドの動作を確認するテスト。

### 統合テスト

複数のコンポーネント間の連携を確認するテスト。

### 回帰テスト

既存機能が変更で破壊されていないかを確認するテスト。

## 開発プロセス関連用語

### エージェント分離

仕様作成、設計、実装、レビュー、テストなど、各責務を異なるエージェントが担当すること。

### ドキュメント駆動開発

設計やテストをドキュメント（仕様、設計書）に基づいて実施する開発方法。

### レビュー駆動開発

複数の観点からのレビューを重視して、品質を確保する開発方法。

## ステータス関連用語

### 進行中（In Progress）

作業が開始されていて、完了していない状態。

### 未作成（Not Created）

未だに作成されていない状態。

### TBD（To Be Determined）

確定していない項目。確定後に更新される予定。

### 承認（Approved）

利害関係者またはレビュアーが承認した状態。
