# code-check

Claude Code を活用して、ソフトウェア開発における仕様作成、設計、実装、コードレビュー、テスト、ドキュメント更新を支援するためのリポジトリです。

本リポジトリでは、Claude Code のカスタムエージェントを利用し、各作業の責務を分離した開発フローを構築します。

## 目的

このリポジトリの目的は、Claude Code を利用した開発作業を標準化し、以下を実現することです。

* 要求やアイデアの整理
* 機能仕様書の作成
* 技術設計書の作成
* 実装計画の作成
* コードの実装支援
* コードレビュー
* コード改善
* テストの作成と評価
* ドキュメントとコードの整合性確認
* Pull Request の説明文作成

## 利用環境

* Claude Code ブラウザ版
* GitHub
* GitHub と Claude Code の連携設定済み

ローカル版 Claude Code の利用は前提としていません。

## 想定する開発フロー

```text
要求整理
  ↓
仕様書作成
  ↓
仕様レビュー
  ↓
設計書作成
  ↓
設計レビュー
  ↓
実装計画
  ↓
実装
  ↓
コードレビュー
  ↓
テスト
  ↓
ドキュメント更新
  ↓
Pull Request 作成
```

仕様、設計、実装、レビュー、テストは、それぞれ異なる役割として扱います。

### `.claude/agents/`

Claude Code のカスタムエージェントを配置します。

予定しているエージェントは以下です。

| エージェント                   | 役割                  |
| ------------------------ | ------------------- |
| `requirements-analyst`   | 要求の整理、前提・制約・未決事項の抽出 |
| `spec-writer`            | 機能仕様書の作成・更新         |
| `design-writer`          | 技術設計書の作成・更新         |
| `implementation-planner` | 実装対象と実装順序の整理        |
| `code-reviewer`          | コード差分のレビュー          |
| `code-improver`          | 可読性、保守性、安全性の改善      |
| `test-writer`            | テストの作成、追加、評価        |
| `security-reviewer`      | 認証、認可、入力検証などの確認     |
| `docs-maintainer`        | コードとドキュメントの整合性確認    |
| `pr-writer`              | Pull Request の説明文作成 |

### `.claude/rules/`

領域別の開発ルールを配置します。

例：

```text
.claude/rules/
├── frontend.md
├── backend.md
├── database.md
└── tests.md
```

### `.claude/skills/`

複数の手順をまとめた、再利用可能な作業フローを配置します。

例：

```text
.claude/skills/
├── create-feature/
├── fix-bug/
└── prepare-pr/
```

### `docs/specs/`

システムや機能が「何をするか」を定義する仕様書を配置します。

仕様書には、原則として以下を含めます。

* 目的
* 対象範囲
* 利用者・アクター
* 機能要件
* 業務ルール
* 正常フロー
* 例外フロー
* 入力検証
* 権限
* エラー処理
* 受入条件
* 対象外
* 未決事項

### `docs/design/`

仕様を「どのように実現するか」を定義する技術設計書を配置します。

設計書には、原則として以下を含めます。

* 設計概要
* アーキテクチャ
* コンポーネント構成
* データモデル
* API
* 処理フロー
* エラー処理
* 認証・認可
* セキュリティ
* 性能
* ログ・監視
* テスト方針
* 移行方針
* リスク

### `docs/adr/`

重要な技術判断を Architecture Decision Record として記録します。

対象例：

* 使用する言語やフレームワーク
* データベース
* 認証方式
* API方式
* リポジトリ構成
* 外部サービス
* 後から変更しにくい設計判断

### `CLAUDE.md`

Claude Code がプロジェクト内で従う共通ルールを記載します。

主な内容は以下です。

* プロジェクト概要
* 仕様書・設計書の保存場所
* 開発手順
* コーディング規約
* テスト方針
* ドキュメント更新方針
* 禁止事項
* 検証コマンド

個別機能の詳細仕様や一時的な作業内容は、`CLAUDE.md` ではなく、対応する仕様書や設計書に記載します。

## ドキュメントの責務

| ドキュメント                  | 責務                   |
| ----------------------- | -------------------- |
| `README.md`             | リポジトリの概要と利用方法        |
| `CLAUDE.md`             | Claude Code が従う共通ルール |
| `docs/project-brief.md` | プロジェクトの目的、利用者、スコープ   |
| `docs/specs/`           | 機能の振る舞いと受入条件         |
| `docs/design/`          | 技術的な実現方法             |
| `docs/adr/`             | 重要な技術判断と理由           |
| `.claude/agents/`       | 専門エージェントの定義          |
| `.claude/rules/`        | 領域別ルール               |
| `.claude/skills/`       | 再利用可能な作業フロー          |

## 基本方針

### 推測で仕様を確定しない

不明な業務ルールや要件は、Claude Code に推測させず、`TBD` または未決事項として記録します。

### 仕様と設計を分離する

仕様書には利用者から見た振る舞いを記載し、具体的な実装方法は設計書に記載します。

### 実装前に計画する

複数ファイルに影響する変更では、実装前に影響範囲、変更対象、テスト方針を整理します。

### レビュー担当はコードを変更しない

コードレビューとコード修正を分離し、レビュー結果を確認してから修正します。

### テストを成功させるために実装を歪めない

テスト担当は、テストを通すために本体コードを変更したり、アサーションを弱めたりしません。

### 実行していない検証を成功と報告しない

テスト、lint、ビルドなどについて、実際に実行した結果のみを記録します。

### コードとドキュメントを同期する

仕様や設計に影響する変更では、実装と同時に関連ドキュメントを確認します。

## Claude Code での利用開始

Claude Code ブラウザ版で本リポジトリを選択し、新しいセッションを開始します。

最初の依頼例：

```text
このリポジトリのREADME.mdを確認してください。

次の初期構成を作成してください。

- CLAUDE.md
- docs/project-brief.md
- docs/development-workflow.md
- docs/glossary.md
- docs/specs/
- docs/design/
- docs/adr/
- .claude/agents/
- .claude/rules/
- .claude/skills/

条件:

- 不明な事項は推測せずTBDとする
- アプリケーションコードはまだ実装しない
- 各ディレクトリの役割を明記する
- 作成したファイルと変更内容を要約する
```

## 新機能開発の依頼例

### 要求整理

```text
requirements-analystを使用してください。

機能要求を整理し、次を提示してください。

- 目的
- 利用者
- 機能要件
- 非機能要件
- 制約
- 前提
- 対象外
- 未決事項
- リスク

不明な業務ルールは推測しないでください。
```

### 仕様書作成

```text
spec-writerを使用してください。

整理済みの要求から機能仕様書を作成し、
docs/specs/配下に保存してください。

機能要件にはFR、業務ルールにはBR、
受入条件にはACのIDを付けてください。

コードは実装しないでください。
```

### 設計書作成

```text
design-writerを使用してください。

承認済みの仕様書に対応する技術設計書を作成し、
docs/design/配下に保存してください。

仕様に存在しない機能を追加しないでください。
コードは実装しないでください。
```

### 実装計画

```text
implementation-plannerを使用してください。

仕様書、設計書、既存コードを確認し、
実装計画を作成してください。

次を含めてください。

- 作成するファイル
- 変更するファイル
- 実装順序
- 完了条件
- 必要なテスト
- 検証コマンド
- リスク

まだファイルを変更しないでください。
```

### コードレビュー

```text
code-reviewerを使用してください。

現在のブランチとmainブランチとの差分をレビューしてください。

次の観点で確認してください。

- 仕様適合性
- 設計適合性
- 正しさ
- 可読性
- 保守性
- エラー処理
- セキュリティ
- 性能
- 後方互換性
- テスト不足

指摘をBLOCKER、HIGH、MEDIUM、LOW、
SUGGESTIONに分類してください。

コードは変更しないでください。
```

### テスト

```text
test-writerを使用してください。

仕様書の受入条件と現在の実装を確認し、
不足しているテストを追加してください。

次を確認してください。

- 正常系
- 異常系
- 境界値
- 入力検証
- 権限
- エラー処理
- 回帰リスク

テストを通すために本体コードを変更しないでください。
```

## Pull Request の推奨単位

```text
PR 1: リポジトリ初期構成
PR 2: CLAUDE.mdと開発ルール
PR 3: カスタムエージェント
PR 4: プロジェクト概要
PR 5: 機能仕様書
PR 6: 技術設計書・ADR
PR 7: 実装
PR 8: テスト・ドキュメント更新
```

変更規模が小さい場合は統合しても構いませんが、仕様、設計、実装を一つの巨大な Pull Request にまとめないことを推奨します。
