# CLAUDE.md

Claude Code がこのプロジェクトで従う共通ルールを定義します。

個別機能の詳細仕様や一時的なタスクは、このファイルではなく対応する仕様書・設計書に記載します。

## プロジェクト概要

**プロジェクト名**: code-check

**目的**: Claude Code を活用して、ソフトウェア開発における仕様作成、設計、実装、コードレビュー、テスト、ドキュメント更新を支援するためのリポジトリ

詳細は `docs/project-brief.md` を参照してください。

## ドキュメント構成

| ドキュメント | 保存場所 | 責務 |

| 技術判断記録 | `docs/adr/` | 重要な技術判断と理由 |

## 実装前に確認する資料

新しい機能や変更を実装する前に、以下を確認してください：

1. **仕様書** (`docs/specs/`) - その機能の受入条件や入力検証、エラー処理を確認
2. **設計書** (`docs/design/`) - 実装方法、API、データモデルなどの詳細設計を確認
3. **技術判断記録** (`docs/adr/`) - 過去の重要な判断に影響しないか確認
4. **領域別ルール** (`.claude/rules/`) - 該当領域のコーディング規約を確認

複数ファイルに影響する変更では、実装前に影響範囲、変更対象、テスト方針を整理してください。

## ドキュメント更新方針

### 同期が必要な場合

- 仕様に影響する変更 → 関連する仕様書を確認・更新
- 設計に影響する変更 → 関連する設計書を確認・更新
- 技術判断に影響する変更 → 関連する ADR を確認・更新

### ドキュメント保存場所

## 基本方針

### 推測で仕様を確定しない

不明な業務ルールや要件は、Claude Code に推測させず、`TBD` または未決事項として記録します。

### 仕様と設計を分離する

- **仕様書**: 利用者から見た振る舞いと受入条件を記載
- **設計書**: 具体的な実装方法、アーキテクチャ、コンポーネント構成を記載

### 実装前に計画する

複数ファイルに影響する変更では、実装前に以下を整理してください：

- 影響範囲
- 変更対象ファイル
- 実装順序
- テスト方針
- リスク

### 役割を分離する

- **コードレビュー**: 指摘者はコードを変更しない
- **テスト**: テスト作成者はテストを通すために本体コードを変更しない
- **ドキュメント作成**: 仕様と設計を同じ人が作成しない

### 実行した検証を記録する

テスト、lint、ビルドなどについて、実際に実行した結果のみを記録してください。

実行していない検証を成功と報告しないでください。

## 禁止事項

以下の行動は禁止です：

- 不明な要件を推測で実装する
- 仕様なしで設計書を作成する
- 設計なしでコードを実装する
- 受入テストなしで実装を完了とする
- テストを通すために本体コードを変更する
- 実行していない検証を成功と報告する

## 不明事項の扱い

不明な事項が発生した場合：

1. **推測しない**: Claude Code は推測で仕様を確定しません
2. **記録する**: `TBD` または **未決事項** として明記します
3. **判断を遅延させる**: 確定が必要になるまで決めずに進めます
4. **報告する**: 実装者に報告し、判断を仰ぎます

例：

```markdown
- 機能 A の権限チェック方法: TBD
- エラーメッセージの多言語対応: 未決事項（後で決定予定）
- キャッシュ有効期限: 確認待ち
```

## カスタムエージェント

このプロジェクトでは、以下のカスタムエージェントを定義しています。

詳細は `.claude/agents/` を参照してください。

| エージェント | 役割 |
|-----------|------|
| `requirements-analyst` | 要求の整理、前提・制約・未決事項の抽出 |
| `spec-writer` | 機能仕様書の作成・更新 |
| `design-writer` | 技術設計書の作成・更新 |
| `implementation-planner` | 実装対象と実装順序の整理 |
| `code-reviewer` | コード差分のレビュー |
| `code-improver` | 可読性、保守性、安全性の改善 |
| `test-writer` | テストの作成、追加、評価 |
| `security-reviewer` | 認証、認可、入力検証などの確認 |
| `docs-maintainer` | コードとドキュメントの整合性確認 |
| `pr-writer` | Pull Request の説明文作成 |
